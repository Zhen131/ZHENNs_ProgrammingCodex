# 编程知识入库指南

> 写给维护 `编程宝典` 的 AI 助手。
> 你的角色不是“多写几个 Markdown 文件”，而是维护一个能持续生长的 Obsidian 编程知识图谱。

## 一、知识库目标

`编程宝典` 用来记录 Ethan 的编程学习过程，并沉淀成可复用的技术知识网络。

它要同时满足两个目标：

- 学习：用发展的眼光理解技术，知道它出现前的痛点、出现后的改变、核心机制和局限。
- 写代码：成为项目开发时可查、可回忆、可维护的个人技术小抄。

## 二、资料库结构

```text
编程宝典/
├── 00 工具与指南/
├── 01 知识图谱入口/
├── 10 计算机基础/
├── 20 编程语言/
├── 30 软件工程/
├── 40 Web与网络/
├── 50 数据与存储/
├── 60 AI与编程工具/
├── 70 项目学习记录/
├── 80 问题与痛点/
└── 90 素材与临时笔记/
```

文件夹用于降低查找成本，`[[wikilink]]` 和 `tags` 用于建立知识图谱。不要只依赖文件夹，也不要把所有文件平铺在根目录。

## 三、三类核心卡片

### 1. 技术卡片

技术卡片是主干，记录稳定概念。

适合建立技术卡片的内容：

- 技术概念：`Closure（闭包）`、`Virtual DOM（虚拟DOM）`
- 编程语言机制：`Class（类）`、`Promise（承诺）`
- 协议和系统：`TCP`、`HTTP`、`Process（进程）`
- 工程方法：`Unit Test（单元测试）`、`Dependency Injection（依赖注入）`
- AI 技术：`Embedding（嵌入）`、`Tokenization（分词）`

技术卡片命名：

```text
English Term（中文翻译）.md
```

示例：

```text
Virtual DOM（虚拟DOM）.md
Waterfall Model（瀑布模型）.md
Tokenization（分词）.md
```

### 2. 问题卡片

问题卡片是入口，记录技术背后的痛点。

适合建立问题卡片的内容：

- `为什么需要 Virtual DOM（虚拟DOM）`
- `为什么异步代码会产生 Callback Hell（回调地狱）`
- `为什么 AI 写的代码容易变成 Black Box（黑盒）`
- `为什么数据库查询会变慢`

问题卡片不要代替技术卡片。它应该链接到多个技术卡片，解释这些技术分别解决了问题的哪一部分。

### 3. 项目学习记录

项目学习记录是证据，记录某次真实学习事件。

适合建立学习记录的内容：

- 在账本项目中理解了某个状态管理方案。
- 调试一个 bug 时学会了某个浏览器机制。
- 读 AI 生成代码时发现自己不懂某个架构概念。

命名建议：

```text
YY_MM_DD 项目名 - 学到的主题.md
```

示例：

```text
26_06_29 账本项目 - 理解 Zustand 状态管理.md
```

## 四、语言规则

本知识库采用“中文逻辑 + 英文术语”的写法。

- 用中文解释背景、痛点、推理、类比、学习反思。
- 技术名词优先使用英文。
- 第一次出现术语时，写成 `English（中文）`。
- 后续重复出现时，可以直接使用英文。
- `aliases` 中补充常见中文名、英文缩写和别名。

示例：

```markdown
`Tokenization（分词）` 是把一段文本切成 token 的过程。LLM 不直接理解自然语言字符串，而是处理 token 序列。
```

## 五、技术卡片模板

```markdown
---
type: 技术卡片
status: 草稿
tags: [类型/技术卡片, 领域/待分类, 层级/待判断]
aliases: []
created: YYYY-MM-DD
updated: YYYY-MM-DD
---

# English Term（中文翻译）

## 一句话

用一句话说明它是什么。

## 背景：它出现前的问题

这个技术出现之前，人们遇到什么麻烦？以前怎么解决？为什么不够好？

## 它解决了什么

它真正降低了什么成本，或者让什么事情第一次变得可行？

## 核心机制

用中文解释核心逻辑，关键术语保留英文。

## 典型使用场景

- 场景 1
- 场景 2

## 局限性

它不能解决什么？滥用会带来什么问题？

## 和其他概念的关系

- [[相关技术卡片]]
- [[相关问题卡片]]

## 在项目中的表现

- [[../70 项目学习记录/某次学习记录]]

## 关联

- [[相关节点]]
```

## 八、入库流程

### 第一步：读懂材料

从用户输入、项目代码、调试过程或学习包中提取：

- 学到的技术概念
- 背后的真实问题
- 所属项目或场景
- 相关语言、框架、协议、工具
- 不确定或需要用户确认的地方

### 第二步：搜索旧节点

新建任何文件前，先搜索：

- 英文术语
- 中文翻译
- 常见缩写
- 同义词
- 相关框架或语言名

如果已有相近节点，优先更新旧节点。

### 第三步：判断入库动作

对每个知识点做判断：

- 已有技术卡片：更新它。
- 没有技术卡片，但概念稳定且未来会复用：新建技术卡片。
- 背后有清晰痛点：新建或更新问题卡片。
- 来自具体项目经历：新建或更新项目学习记录。
- 还不成熟：放入 `90 素材与临时笔记`。

### 第四步：建立连接

每个新文件至少要连接到：

- 1 个上位概念或所属领域
- 1 个相关技术、问题或项目记录

不要通过额外导航文件制造间接关系。正文中第一次提到关键概念时，应直接使用 `[[wikilink]]` 连到对应的技术卡片、问题卡片或项目学习记录。

### 第五步：更新旧节点

如果新材料深化了旧概念，在旧节点中追加：

```markdown
## 进一步理解

- [[新节点]]：一句话说明这次材料如何补充了这个概念。
```

### 第六步：检查图谱关系

入库完成后，检查新旧节点之间是否已经直接连接：

- 技术卡片要链接到它依赖、对比或协作的技术卡片。
- 问题卡片要链接到真正解决该痛点的技术卡片。
- 项目学习记录要链接到这次学习沉淀出的技术卡片和问题卡片。
- 如果新材料加深了旧节点，旧节点也要反向链接回来。

不要为了“看起来完整”创建额外导航清单。关系应该长在卡片正文里。

## 九、标签规范

推荐标签维度：

- `类型/技术卡片`
- `类型/问题卡片`
- `类型/项目学习记录`
- `领域/计算机基础`
- `领域/编程语言`
- `领域/软件工程`
- `领域/Web与网络`
- `领域/数据与存储`
- `领域/AI与编程工具`
- `语言/JavaScript`
- `语言/Python`
- `语言/Java`
- `语言/C`
- `层级/基础`
- `层级/进阶`
- `状态/草稿`
- `状态/待深化`
- `状态/常青`

标签表达分类，链接表达关系。不要用标签替代链接。

## 十、入库检查清单

完成入库后确认：

- [ ] 文件放在正确目录。
- [ ] 技术卡片使用英文优先命名。
- [ ] YAML frontmatter 完整。
- [ ] 正文中关键概念已内嵌 `[[wikilink]]`。
- [ ] `## 关联` 不是空的。
- [ ] 搜索过旧节点，没有重复造概念。
- [ ] 相关旧节点已更新。
- [ ] 新旧节点之间已经建立直接链接，没有依赖额外导航文件做中转。
- [ ] 如果内容还不成熟，已放入 `90 素材与临时笔记` 而不是强行沉淀。
