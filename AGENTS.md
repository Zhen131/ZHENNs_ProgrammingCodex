# AGENTS.md

本仓库是 Ethan 的个人 Obsidian 编程知识库：`编程宝典`。

它不是普通资料夹，而是一个长期生长的技术知识图谱。所有 AI 助手在修改本仓库前，必须先理解这里的入库规则。

## 必读文件

修改本知识库前，先读：

- `00 工具与指南/编程知识入库指南.md`
- `00 工具与指南/skills/programming-knowledge-ingest/SKILL.md`

如果任务发生在其他项目中，目的是整理学习过程、调试过程、读代码过程或 AI 编程过程，先读：

- `00 工具与指南/skills/programming-learning-summarizer/SKILL.md`

## 核心原则

1. 技术卡片是主干。
   - 稳定技术概念、编程语言特性、框架机制、协议、工程方法都应沉淀成技术卡片。

2. 问题卡片是入口。
   - 用真实痛点解释技术为什么出现，例如“为什么需要 Virtual DOM（虚拟DOM）”。

3. 不制造学习中转层。
   - 项目经历只能作为技术卡片或问题卡片正文中的语境和证据，不要单独生成聚合式学习文件。

4. 不制造孤岛文件。
   - 新增文件时必须建立 `[[wikilink]]`。
   - 发现相关旧节点时，必须更新旧节点的关联、引用或“进一步理解”部分。
   - 优先让技术卡片和问题卡片彼此直接连接，不依赖额外导航文件做中转。

5. 中文讲逻辑，英文定术语。
   - 解释、背景、推理、反思用中文。
   - 技术名词优先用英文，第一次出现时补中文：`Embedding（嵌入）`、`Tokenization（分词）`。
   - 技术卡片文件名采用英文优先：`Virtual DOM（虚拟DOM）.md`。

6. 先查重，再新建。
   - 新建节点前，用文件名、别名、关键词搜索现有节点。
   - 如果已有相近节点，优先更新旧节点，不要重复造概念。

## Git 规则

- 不要删除用户已有内容，除非用户明确要求。
- 不要重写历史或执行破坏性 Git 命令。
- 修改完成后，优先给出 `git status` 结果摘要。
- Obsidian 的个人窗口状态不应作为知识内容提交。

## 文件夹定位

- `00 工具与指南`：规则、SKILL、模板、维护说明。
- `01 知识图谱入口`：知识库总览和入库方向说明，不作为卡片中转层。
- `10 计算机基础`：计算机系统、操作系统、算法、数据结构、编译原理等基础。
- `20 编程语言`：语言特性、语法、类型系统、运行时。
- `30 软件工程`：架构、测试、重构、Git、CI/CD、设计模式。
- `40 Web与网络`：HTTP、TCP/IP、浏览器、前后端、Web 框架。
- `50 数据与存储`：数据库、缓存、索引、事务、文件与对象存储。
- `60 AI与编程工具`：LLM、MCP、AI 编程、NLP、Embedding、Tokenization。
- `80 问题与痛点`：以“为什么”“如何解决”为中心的问题卡片。
- `90 素材与临时笔记`：待整理材料、草稿、摘录、临时学习包。
