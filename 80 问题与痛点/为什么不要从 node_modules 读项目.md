# 为什么不要从 node_modules 读项目

## 痛点

拿到 JavaScript 项目时，`node_modules/` 往往是根目录里最显眼、文件最多的目录。初学者容易误以为：既然这里有这么多代码，那是不是都要读懂？

真正的问题是：把第三方依赖目录当成当前项目源码，会让读项目的入口彻底失焦。

## 以前怎么解决

如果没有明确的工程分层意识，初学者可能会直接翻 `node_modules/`，试图从依赖内部实现理解项目。但这会遇到几个问题：

- 文件数量巨大。
- 大部分代码不是当前项目作者写的。
- package 还会依赖其他 package。
- 读完依赖也不一定知道业务代码在哪里。

## 技术演进

现代 JavaScript 项目用 [[npm（Node 包管理器）]] 管理 [[Package（软件包）]]。项目通过 [[package.json（项目清单）]] 声明依赖，npm 再把依赖实体安装到 [[node_modules（依赖实体目录）]]。

所以读项目的正确入口不是 node_modules，而是：

- README：理解项目目标。
- package.json：理解命令和依赖。
- `src/`：理解业务源码。

## 我的理解

node_modules 像工具仓库，不像施工图纸。工程师需要知道工具仓库存在，也需要知道工具清单在哪里，但不会通过翻完整个工具仓库来理解当前项目。

以后看到 node_modules，我的判断应该是：

```text
这是第三方依赖实体
不是当前项目的业务源码
不读
不手改
通常不提交
```
