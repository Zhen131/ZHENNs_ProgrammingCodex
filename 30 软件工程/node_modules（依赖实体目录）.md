# node_modules（依赖实体目录）

## 一句话

`node_modules（依赖实体目录）` 是 npm 把 package 下载到本地后存放依赖实体的目录。

## 背景：它出现前的问题

初学 JavaScript 项目时，看到 node_modules 中有大量文件夹，很容易以为这些都是必须读懂的项目源码。结果会把精力浪费在第三方依赖内部实现上，反而看不清当前项目自己的业务结构。

## 它解决了什么

node_modules 让项目可以在本地拥有依赖实体，运行和构建时工具能够找到已经安装的 package。

可以这样理解：

```text
package.json   写着我要哪些工具
npm install    负责下载这些工具
node_modules   把工具真正放在本地
```

## 核心机制

node_modules 里不只包含项目直接声明的依赖，还包含依赖的依赖，也就是传递依赖。所以它通常非常大、非常乱。

工程上的判断规则是：

- 不读整个 node_modules。
- 不手改 node_modules。
- 不把 node_modules 当成业务源码。
- 通常不提交 node_modules。
- 想知道项目用了什么库，看 [[package.json（项目清单）]]。
- 想知道某个库在哪里被项目使用，搜索 `src/`。

## 典型使用场景

- npm 安装依赖后的本地存放位置。
- 本地开发服务器和测试工具查找 package 的地方。
- 排查某个依赖是否安装成功时偶尔查看。

## 局限性

node_modules 是安装结果，不是依赖声明。删除后可以通过 npm 重新安装。它也不是 [[Node.js（Node 运行环境）]] 本体，不能把两者混为一谈。

## 和其他概念的关系

- [[npm（Node 包管理器）]] 负责创建和维护 node_modules。
- [[Package（软件包）]] 安装后会进入 node_modules。
- [[Dependency（依赖）]] 解释为什么 node_modules 里会有直接依赖和传递依赖。
- [[为什么不要从 node_modules 读项目]] 是它对应的问题入口。
