# package-lock.json（依赖锁文件）

## 一句话

`package-lock.json（依赖锁文件）` 记录 npm 实际安装的精确依赖版本、下载来源、完整性校验和依赖树，用来提高不同机器安装结果的一致性。

## 背景：它出现前的问题

[[package.json（项目清单）]] 通常只表达项目需要哪些 package 和允许的版本范围。不同时间、不同机器安装依赖时，如果只看版本范围，可能得到不同的依赖树，造成“我这里能跑，你那里不能跑”。

## 它解决了什么

package-lock.json 把“项目需要哪些工具”进一步固定成“这次到底精确安装了哪些版本的工具”，包括依赖的依赖。

可以先这样记：

```text
package-lock.json
= 机器采购清单和精确型号记录
```

它帮助团队尽量保证：

- 本机安装的依赖版本一致。
- 其他开发者安装的依赖版本一致。
- CI/CD 环境安装的依赖版本一致。

## 核心机制

依赖锁文件会记录：

- 直接依赖的精确版本。
- 传递依赖的精确版本。
- 下载来源。
- 完整性校验值。
- 完整依赖树。

它不是运行日志，不记录“什么时候按了 dev、按了几次、运行结果如何”。它记录的是这套机器和零件到底买了哪个精确型号、从哪里下载、校验码是什么，以及附带了哪些子依赖。

通常开发者不需要手动阅读或编辑它，但它应该和 package.json 一起提交到 Git。

## 典型使用场景

- 让 `npm install` 尽量复现同一套依赖。
- 排查依赖版本差异导致的问题。
- 在团队协作和部署环境中固定依赖树。

## 局限性

package-lock.json 不是业务源码，也不是初学读项目的主入口。读项目时应先看 README、[[package.json（项目清单）]] 和 `src/`，遇到依赖版本问题时再深入锁文件。

如果 [[package.json（项目清单）]] 说“我要 Next.js 14.2.35、React 18、decimal.js 10.6.x”，package-lock.json 会把实际装下来的精确结果写清楚，例如具体版本、下载地址、完整性校验值和子依赖。

## 和其他概念的关系

- [[npm（Node 包管理器）]] 生成并使用 package-lock.json。
- [[package.json（项目清单）]] 声明想要哪些依赖，package-lock.json 记录实际精确安装结果。
- [[Dependency（依赖）]] 的精确安装结果会进入锁文件。
- [[node_modules（依赖实体目录）]] 是锁文件描述的依赖树在本地的实体结果。
- [[npm scripts（npm 脚本）]] 执行命令，但命令运行过程不会被 package-lock.json 记录。
- [[JavaScript Project Structure（JavaScript 项目结构）]] 把 package-lock.json 归为“知道用途即可，遇到问题再深入”的配置类文件。
